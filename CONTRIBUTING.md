# Contributing Guide

## Governance

This repository follows an approval-based workflow.

- The CEO approves merges to `main`
- No direct pushes to production
- Changes should be proposed through pull requests
- Each pull request should clearly state purpose, scope, and user-facing impact

## Branch naming

Use clear branch names, for example:

- `feature/homepage`
- `feature/products-section`
- `feature/services-section`
- `feature/contact-section`
- `docs/repo-governance`

## Commit message format

Use concise conventional commit-style messages:

- `feat: add homepage hero section`
- `feat: add product cards for approved initiatives`
- `docs: add Cloudflare deployment guide`
- `fix: update contact CTA copy`

## Pull request checklist

Before requesting review:

- [ ] Content matches approved brand direction
- [ ] No unapproved claims or promises are added
- [ ] Mobile layout has been checked
- [ ] Contact details are correct
- [ ] Documentation is updated if workflow changes
- [ ] Deployment impact is noted

## Content guardrails

Do not introduce the following without explicit approval:

- New product promises
- Public launch dates
- Pricing
- Testimonials
- Security claims
- Performance claims
